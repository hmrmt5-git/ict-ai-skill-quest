# レベル1試作版 実装計画

この計画は、ICT・AI SKILL QUESTを小さな単位で安全に実装するための工程表です。各フェーズの完了条件を満たしてから次へ進みます。

## フェーズ0：設計と開発基盤

1. 目的: 要件、採用技術、開発ルール、テスト方針を明確にする。
2. 実装する機能: README、AGENTS.md、要件定義、技術比較、アーキテクチャ決定、データモデル、セキュリティ設計、CI基盤。
3. 実装しない機能: 実認証、実データ保存、PDF生成本番処理。
4. 作成・変更予定ファイル: `README.md`, `AGENTS.md`, `docs/requirements.md`, `docs/architecture-comparison.md`, `docs/architecture-decision.md`, `docs/data-model.md`, `docs/security-design.md`, `.github/workflows/pages.yml`。
5. 完了条件: 設計文書が揃い、`npm run lint` が通る。
6. 単体テスト: ルール関数の基本テストを準備する。
7. 結合テスト: CIでlint/build/testを順番に実行できることを確認する。
8. セキュリティ確認: 秘密情報をコミットしない設定と `.env.example` のみ管理対象にする。
9. 人が行う外部設定: GitHub Pages有効化、Firebaseプロジェクト作成候補の確認。
10. 次フェーズへ進む判断条件: 主要文書の名称、採用技術、実装範囲に矛盾がない。

## フェーズ1：フロントエンド基盤

1. 目的: GitHub Pagesで動く日本語UIの骨格を作る。
2. 実装する機能: トップページ、レベルカード、レスポンシブCSS、ライト/ダーク、サウンドON/OFF、動きを減らす設定、別タブ導線。
3. 実装しない機能: 実Googleログイン、サーバー保存、実PDF生成。
4. 作成・変更予定ファイル: `public/index.html`, `public/styles/app.css`, `src/app.ts`, `src/styles/app.css`。
5. 完了条件: 直接URLとサブパス配信で画面が表示できる。
6. 単体テスト: テーマ・サウンド保存関数。
7. 結合テスト: 画面遷移、Google Sites iframe検知、別タブリンク。
8. セキュリティ確認: UIに秘密情報や管理者パスワードを書かない。
9. 人が行う外部設定: GitHub Pages URL候補を確認する。
10. 次フェーズへ進む判断条件: Chromebook、PC、スマートフォン相当幅で大きな崩れがない。

## フェーズ2：Firebase等のバックエンド基盤

1. 目的: 本番でサーバー側検証を行う土台を作る。
2. 実装する機能: Firestoreコレクション、Security Rules、Cloud Functions、Storage、環境変数設計。
3. 実装しない機能: 画面の最終デザイン調整。
4. 作成・変更予定ファイル: `firebase.json`, `firestore.rules`, `functions/*`, `.env.example`, `docs/deployment-overview.md`。
5. 完了条件: ローカルエミュレータまたは検証環境でFunctionsとRulesが動く。
6. 単体テスト: ドメイン判定、権限判定、トランザクション処理。
7. 結合テスト: FunctionsからFirestore/Storageへアクセスできる。
8. セキュリティ確認: 秘密鍵、OAuthシークレット、サービスアカウントJSONをコミットしない。
9. 人が行う外部設定: Firebaseプロジェクト作成、Google認証プロバイダ有効化、管理者初期登録。
10. 次フェーズへ進む判断条件: クライアントだけで採点・合否・回数制限を決めない構成になっている。

## フェーズ3：Google認証

1. 目的: `wasegaku.ac.jp` のGoogle Workspaceアカウントだけが利用できるようにする。
2. 実装する機能: Firebase Auth Googleログイン、IDトークン検証、メール確認済み確認、サーバー側ドメイン確認。
3. 実装しない機能: 管理者画面の全機能。
4. 作成・変更予定ファイル: `src/lib/auth.ts`, `functions/auth.ts`, `docs/security-design.md`, `README.md`。
5. 完了条件: 外部ドメインの登録処理がサーバー側で拒否される。
6. 単体テスト: `@wasegaku.ac.jp` 判定、メール確認済み判定。
7. 結合テスト: ログイン、ログアウト、iframe時の失敗表示、別タブ導線。
8. セキュリティ確認: Googleのhd指定だけを信用しない。
9. 人が行う外部設定: Firebase Authenticationの承認済みドメイン設定。
10. 次フェーズへ進む判断条件: 認証済みユーザーIDを全APIで検証できる。

## フェーズ4：利用者登録

1. 目的: 初回ログイン時に必要な自己申告情報を安全に登録する。
2. 実装する機能: 表示用ユーザー名、利用者区分、所属校、重複禁止、30日変更制限、履歴保存。
3. 実装しない機能: 一括ユーザー登録。
4. 作成・変更予定ファイル: `src/features/profile/*`, `functions/users.ts`, `docs/data-model.md`。
5. 完了条件: 同時登録でも同じ正規化ユーザー名を二重登録できない。
6. 単体テスト: ユーザー名正規化、禁止語、文字数、30日制限。
7. 結合テスト: 新規登録、重複エラー、変更履歴。
8. セキュリティ確認: 外部ドメイン利用者は登録APIを実行できない。
9. 人が行う外部設定: 禁止語リスト、所属校選択肢の確認。
10. 次フェーズへ進む判断条件: 登録済みユーザーだけが挑戦開始できる。

## フェーズ5：問題管理と出題

1. 目的: 試作用仮問題20問から安全に10問を出題する。
2. 実装する機能: 問題データ、ICT3/AI6/総合1抽出、未出題優先、選択肢ランダム化、管理者プレビュー。
3. 実装しない機能: レベル2〜5本番問題、全250問。
4. 作成・変更予定ファイル: `src/data/questions.ts`, `functions/questions.ts`, `docs/question-import-format.md`。
5. 完了条件: クライアントへ正解・得点を事前配信しない。
6. 単体テスト: 分野別問題数、未出題優先、選択肢ランダム化。
7. 結合テスト: 挑戦開始時の問題セット作成、複数タブ二重発行防止。
8. セキュリティ確認: 問題得点・非公開解説を一般APIで返さない。
9. 人が行う外部設定: 本番問題レビュー、試作用ラベルの確認。
10. 次フェーズへ進む判断条件: 問題セットが履歴に保存され、再読み込みで再開できる。

## フェーズ6：回答、採点、履歴

1. 目的: 回答確定、サーバー側採点、履歴保存、結果表示を実装する。
2. 実装する機能: 回答API、各問題1回のみ回答、時間切れ自動提出、84点合格、分野別得点、履歴表示。
3. 実装しない機能: 外部生成AIによるアドバイス。
4. 作成・変更予定ファイル: `functions/attempts.ts`, `src/features/quest/*`, `src/lib/rules.ts`。
5. 完了条件: 連続クリック・再送信でも同一問題が二重回答されない。
6. 単体テスト: 10/8/5/2/0点、84点合格、83点不合格、分野別集計。
7. 結合テスト: 挑戦開始、回答、タイマー終了、結果保存。
8. セキュリティ確認: 採点と合格判定をクライアントだけで行わない。
9. 人が行う外部設定: なし。
10. 次フェーズへ進む判断条件: 不合格時も肯定的なアドバイスが出る。

## フェーズ7：サウンドと合格演出

1. 目的: 学校で使いやすい控えめな音と、合格時の達成感を提供する。
2. 実装する機能: 得点別Web Audioサウンド、サウンドOFF初期値、くす玉、紙吹雪、垂れ幕、ファンファーレ、スキップ、動きを減らす設定。
3. 実装しない機能: 外部著作権音源の利用。
4. 作成・変更予定ファイル: `src/features/sound/*`, `src/features/effects/*`, `public/styles/app.css`。
5. 完了条件: サウンドOFFで全音が停止し、視覚表示だけで結果が分かる。
6. 単体テスト: サウンド設定保存、得点別音種別選択。
7. 結合テスト: 回答後音、合格後演出、スキップ、動きを減らす設定。
8. セキュリティ確認: 外部音源URLや不要なトラッキングを入れない。
9. 人が行う外部設定: なし。
10. 次フェーズへ進む判断条件: アクセシビリティを損なわない。

## フェーズ8：修了証とQRコード

1. 目的: 合格記録を証明できる修了証を安全に発行する。
2. 実装する機能: PDF生成、修了証番号、合格時点スナップショット、伏字メール、QR検証URL、無効化。
3. 実装しない機能: 本番ロゴ作成、レベル5特別バッジ完成版。

7. 結合テスト: 合格処理、証明書発行、QR検証、無効化反映。

10. 次フェーズへ進む判断条件: 合格者数更新と証明書発行の整合性が保てる。

2. 実装する機能: システム管理者、問題管理者、閲覧管理者、利用者検索、問題管理、修了証管理、集計、監査ログ。
